# 1. 이력서 기반 핵심 강점 요약

## 1.1 한 문장 포지셔닝

**저는 Elasticsearch를 쿼리 작성 도구로만 사용한 개발자가 아니라, 검색 서비스의 클러스터·색인 파이프라인·형태소 사전·검색 쿼리·랭킹·모니터링·장애 대응까지 End-to-End로 운영하고 개선해 온 검색엔지니어입니다.**

## 1.3 기술 영역별 핵심 경험

### Elasticsearch·검색

- Elasticsearch 클러스터 구축, 운영, 버전 업그레이드, 보안, RBAC, 노드·리소스 튜닝
- 상품·브랜드·카테고리 검색 운영
- Nori 형태소 분석기, 동의어·불용어·사용자 사전, `edge_ngram`, Analyzer·Mapping 운영
- Java/JPA 기반 정적·증분 색인 파이프라인 유지보수
- Mustache Template 기반 구조를 Query DSL 코드 조합 방식으로 전환
- `function_score`, `boost`, `random_score`를 활용한 채용관 랭킹 개선
- 검색 템플릿 Git·Jenkins 배포 자동화
- Windows 기반 Elasticsearch 클러스터를 Linux로 무중단성에 가깝게 이관하고 데이터 정합성 검증
- Rust 기반 사전 배포·동기화 및 로그·메트릭 수집·알림 시스템 구축

### Kafka·데이터 파이프라인

- 처리량을 고려한 Kafka 토픽·파티션·복제 전략 설계
- Kafka 버전 업그레이드와 Kafka-UI·CMAK 모니터링
- API→Kafka→Elasticsearch 그룹사 통합 로깅 파이프라인 설계·구축
- Kafka→Logstash→Elasticsearch 구조를 Kafka→Vector→Elasticsearch로 단순화
- 버퍼링, 변환, 필터링, 처리량 벤치마크 경험

### RDB

- SQL Server Stored Procedure 실행계획 분석, 쿼리 재작성, Non-Clustered Index 설계
- ERD 기반 정규화·스키마 설계
- Timeout·Schema 변경 모니터링 및 장애 알림
- 그룹사별 개인정보 저장 구조 분석과 Rust 기반 파기 스케줄러 구축
- MySQL은 현재 기술 스택과 개인 프로젝트에서 활용하고 있으며, 실무 핵심 강점은 SQL Server 운영·튜닝 경험이다.

### Rust·Linux·운영 자동화

- Elasticsearch 사전 관리, 템플릿 배포 CLI, 로그·메트릭 수집, 개인정보 파기 스케줄러를 Rust로 구현
- Ubuntu에서 Elasticsearch·Kafka·Redis·MongoDB 설치와 클러스터 구성 자동화
- Linux ES 클러스터 신규 구축, `systemd` 서비스·로그 운영 체계 구성
- 장애 징후를 Telegram·사내 메신저로 알리는 모니터링 시스템 구축

## 1.4 숫자로 말할 수 있는 대표 성과

- 검색 템플릿 배포 시간 90% 이상 단축
- Elasticsearch 사전 업데이트 30분→9분, 수동 오류율 90% 감소
- 검색엔진 장애 대응 시간 30분→10분
- Windows→Linux 전환 후 검색 성능 약 15% 개선, 서버 라이선스 비용 약 66% 절감
- 채용관 유료공고 노출량 평균 25% 증가, 응답속도 최대 15% 개선
- Kafka→Vector 전환으로 메모리 사용량 40% 감소, 로그 처리 속도 약 60% 향상

> 숫자를 말하면 반드시 측정 기간, 비교 기준, 모니터링 도구, 표본을 이어서 설명할 수 있어야 한다. 면접 전 실제 근거를 다시 확인한다.

## 1.6 면접에서 주의할 약점과 대응

### OpenSearch 직접 경험

과장해서 “경험이 있다”고 말하지 않는다.

> “OpenSearch를 운영 환경에서 직접 담당한 경험은 아직 없습니다. 다만 Elasticsearch 클러스터 운영, 버전 업그레이드, Mapping·Analyzer·Query DSL·색인 파이프라인·플러그인 호환성 검증을 해왔기 때문에 전환 시 확인해야 할 지점을 이해하고 있습니다. 합류하게 되면 API와 플러그인 차이, 쿼리·스코어 정합성, 성능 벤치마크를 먼저 정리하고 작은 트래픽부터 단계적으로 이전하겠습니다.”

### 다국어 형태소 분석

직접 강점은 한국어 Nori·사전 운영이다.

> “실무의 중심은 한국어 형태소 분석이었고 다국어 분석기를 깊게 운영한 경험은 제한적입니다. 대신 형태소 오류를 로그와 검색 결과로 확인하고 사용자 사전과 Analyzer를 개선하는 운영 루프는 경험했습니다. 특허 데이터에서는 언어별 Analyzer뿐 아니라 원문·번역문·정규화 필드 전략이 중요할 것으로 보고, 도메인 전문가와 평가셋을 만들면서 확장하겠습니다.”

### 특허 검색식 파서

프로덕션 파서 개발 경험은 없다고 명확히 한다.

> “검색식 파서를 직접 운영한 경험은 없지만, 복잡한 검색 조건을 Query DSL 구조로 전환하고 검증한 경험은 있습니다. 파서를 맡게 된다면 토큰화, 문법 분석, AST, 의미 검증, Query DSL 변환을 분리하고, 중첩 괄호·필드 지정·근접 연산·잘못된 입력을 테스트 케이스로 관리하겠습니다.”

### Kafka 표현

Kafka를 NoSQL이라고 부르지 않는다. **분산 이벤트 스트리밍 플랫폼 또는 메시징·데이터 파이프라인**으로 표현한다.

### 대용량의 근거

이력서에는 노드 수, 인덱스 용량, 문서 수, QPS, 일 색인량이 없다. 아래 숫자를 면접 전 반드시 채운다.

# 2. 워트인텔리전스 회사 분석

## 2.1 확인된 사실

### 사업과 미션

- 워트인텔리전스는 2015년에 시작한 특허·기술 데이터 기반 AI 기업이다.
- 회사는 복잡하고 전문적인 특허 정보를 소수 전문가의 전유물이 아닌 모두의 전략 자산으로 만드는 것을 미션으로 제시한다.
- 공식 채용 페이지는 전 세계 106개국에서 수집한 3억 건의 특허 데이터를 기반으로 AI 특허 리서치 환경을 만든다고 설명한다.
- 공식 홈페이지는 3,000개 이상의 파트너와 함께한다고 밝히고 있다.

### 주요 제품

- **키워트(keywert)**: 글로벌 AI 특허 검색·분석 엔진. 공식 페이지 기준 3억 건 이상의 특허 문헌 검색, 주요 15개국 상세 정보와 그 외 91개국 초록 검색, 검색식 작성·분류·시각화·협업 기능을 제공한다.
- **키워트 인사이트**: 자연어 질의를 바탕으로 검색을 넘어 분석 결과와 인사이트를 만드는 AI Native Research 제품이다.
- **IP 데이터 인텔리전스**: 특허 빅데이터 분석 기반 비즈니스 컨설팅·분석 서비스다.
- **IP 킹콩**: 특허 전문 AI 번역 플랫폼이다.
- **팻스푼**: 특허 전문 교육 플랫폼이다.
- **디시전 인텔리전스**: 자산실사, 기술이전 후보 추천, 기술가치평가 등을 지원하는 AI 기반 기술사업화 솔루션이다.

### 데이터·AI 기술 방향

- 자체 특허 특화 언어모델 `PlutoLM`과 하이브리드 검색 아키텍처를 공개했다.
- 공개 자료상 PlutoLM은 1억 7천만 건의 특허, 2,500억 개 특허 문장, 16억 장의 특허 도면, 1억 5천만 건의 가공 데이터를 활용했다.
- LG AI연구원과 특허 특화 AI 모델을 공동 개발하고, DeepL·리벨리온 등과 번역·AI 인프라 영역에서 협력을 공개했다.
- 회사는 기존의 정밀 특허 검색뿐 아니라 자연어 기반 AI 리서치, 온프레미스·보안, 일본을 포함한 글로벌 시장 확장을 강조하고 있다.

### 검색엔지니어 채용 포지션

공개 공고의 주요 업무는 다음과 같다.

- ES8로 스키마·데이터·검색 쿼리 이전
- 데이터 파이프라인 작성·운영
- ES8에서 OpenSearch로 기능 이전
- 검색 클러스터 모니터링, 장애 관리, 성능 개선과 재발 방지
- 키워트 인사이트 등 AI 제품에 필요한 특허 데이터 가공·인덱싱
- 특허 검색식 파서 개발·유지보수

요구 역량은 Elasticsearch·OpenSearch·Solr 운영, 대용량 검색 시스템, 한국어·다국어 형태소 분석, 형태소 오류 교정과 사전 운영, Python·Java다. Vector Search·RAG, 데이터 기반 파라미터 최적화, 레거시 리팩터링, 복잡한 대규모 쿼리 최적화가 우대사항이다.

### 조직 문화·채용

공개 자료 기준 대표는 윤정호 CEO이며, 기술총괄은 나호열 CTO다. 나호열 CTO는 공개 행사에서 PlutoLM, 하이브리드 검색 아키텍처, 데이터 보안과 전문가 참여형 평가 구조를 설명했다.

공식 채용 페이지가 제시한 일하는 방식은 다음 네 가지다.

1. **문제는 구조화부터**: 전체를 빠짐없이, 중복 없이, 논리적으로 본다.
2. **진짜 원인 찾기**: “왜 그런가, 정말 그런가”를 묻고 끝까지 파고든다.
3. **먼저 맞추고 시작하기**: 왜·누가·언제까지·얼마만큼 할지 합의한다.
4. **논쟁하되 따르기**: 토론하고 결정한 뒤 동의하고 실행한다.

공식 채용 절차는 서류→직무 인터뷰→컬처핏 인터뷰→처우 협의 순서다. 따라서 이번 2차 면접은 명칭상 컬처핏 인터뷰에 해당할 가능성이 높다.

## 2.2 합리적인 추정

아래는 공개 자료에 기반한 추정이며 내부 사정으로 단정하면 안 된다.

- 대규모 레거시 검색 시스템을 ES8과 OpenSearch로 옮기는 과제가 실제 우선순위일 가능성이 높다.
- 키워트 인사이트의 성장으로 전통적인 Boolean·필드 검색과 벡터·LLM 기반 검색을 함께 운영하는 하이브리드 검색 역량이 중요해질 가능성이 높다.

- CEO는 검색 기술 자체보다 그것이 고객의 리서치 시간, 정확도, 제품 경쟁력, 글로벌 확장에 어떻게 기여하는지를 확인할 가능성이 높다.
- CTO는 화려한 기술 키워드보다 마이그레이션 리스크 관리, 장애 원인 규명, 검색 품질 검증, 데이터 정합성 같은 운영 기본기를 확인할 가능성이 높다.

# 4. CTO 예상 질문 및 답변

## Q1. 본인을 검색엔지니어라고 정의하는 이유는 무엇인가요?

> “검색엔진을 단순히 API로 호출한 것이 아니라 검색 결과가 만들어지고 안정적으로 제공되는 전 과정을 운영했기 때문입니다. 위메프에서는 상품·브랜드·카테고리 검색과 Nori·사용자 사전, 정적·증분 색인 파이프라인을 맡았고, 알바천국에서는 Query DSL 구조 개선, 랭킹, 사전 배포, 클러스터 이관과 모니터링을 담당했습니다. 그래서 저는 검색 품질과 클러스터 안정성, 데이터 정합성을 하나의 시스템으로 보는 편입니다. 워트에서도 특허 데이터가 들어오는 단계부터 사용자가 결과를 신뢰하는 단계까지 책임지는 검색엔지니어로 일하고 싶습니다.”

## Q2. 가장 어려웠던 기술적 전환은 무엇이었나요?

> “Windows 기반 Elasticsearch 클러스터를 Linux로 이관한 경험이 가장 대표적입니다. 새 클러스터를 만드는 것보다 기존 검색 결과와 색인 데이터가 그대로 유지되는지를 검증하는 일이 더 중요했습니다. 그래서 Mapping과 Analyzer, 검색 템플릿의 호환성을 먼저 확인하고, 정적·증분 색인의 건수와 샘플 데이터를 비교했습니다. 운영 전에는 주요 검색 시나리오의 결과와 응답시간을 비교했고, 이관 후에는 Kibana와 시스템 메트릭을 함께 보면서 안정화했습니다. 결과적으로 데이터 손실 없이 이관했고 성능은 약 15%, 라이선스 비용은 약 66% 개선했습니다. 이 경험을 워트의 ES8·OpenSearch 이전에서도 기능 정합성과 운영 리스크를 관리하는 방식으로 활용할 수 있습니다.”

## Q3. ES8에서 OpenSearch로 이전한다면 어떻게 접근하겠습니까?

> “한 번에 전환하기보다 호환성 목록과 검증 기준부터 만들겠습니다. 먼저 사용 중인 Mapping, Analyzer, 플러그인, Query DSL, 집계, 스크립트, 클라이언트 API를 목록화하고 OpenSearch와의 차이를 분류하겠습니다. 그다음 동일 데이터셋과 대표 쿼리셋으로 결과 건수, 상위 문서 순서, 스코어, P95 응답시간, 색인 처리량을 비교하겠습니다. 데이터는 재색인 가능한 경로를 확보하고 alias나 이중 쓰기 방식을 검토하며, 읽기 트래픽을 소량부터 옮기겠습니다. 직접 OpenSearch를 운영한 경험은 없지만, Elasticsearch 이관과 버전·호환성 검증을 해왔기 때문에 리스크를 구조화하고 단계적으로 전환하는 방식에는 자신이 있습니다.”

## Q4. 검색 성능이 갑자기 떨어졌을 때 무엇부터 보나요?

> “먼저 사용자 영향과 변경 이력을 확인하고, 검색 계층과 인프라 계층을 나눠 범위를 줄입니다. P95·P99 응답시간, 오류율, 검색량 변화와 함께 CPU, heap, GC, disk I/O, thread pool rejection, shard 상태, query cache를 봅니다. 특정 쿼리에서만 느리다면 slow log와 profile API로 필터·정렬·집계·스크립트 비용을 확인하고, 전체가 느리다면 노드·샤드·GC·I/O를 우선 의심합니다. 우선 영향도를 줄이기 위해 최근 변경 롤백이나 무거운 기능 제한을 판단하고, 복구 뒤에는 원인과 재현 조건을 남겨 알림 임계치나 테스트에 반영합니다. 장애를 빨리 끄는 것과 다시 발생하지 않게 만드는 것을 한 작업으로 봅니다.”

## Q5. 모니터링에서 가장 중요한 지표는 무엇인가요?

> “한 지표보다 사용자 지표와 원인 지표를 연결하는 것이 중요합니다. 사용자 관점에서는 검색 성공률, 오류율, P95·P99 응답시간을 보고, 원인 지표로 heap·GC, CPU, disk 사용률과 I/O, shard 상태, thread pool queue·rejection, query cache, 색인 지연을 봅니다. 위메프와 알바천국에서는 Rust, Metricbeat, Heartbeat, Grafana, Kibana를 이용해 이런 지표를 수집하고 임계치를 넘으면 메신저와 Telegram으로 알리도록 구성했습니다. 그 결과 검색 장애 대응 시간을 약 30분에서 10분으로 줄였습니다. 워트에서도 단순 서버 모니터링이 아니라 검색 품질과 데이터 최신성까지 이어지는 관측 체계를 만들고 싶습니다.”

## Q6. Mustache Template에서 Query DSL 구조로 바꾼 이유는 무엇인가요?

> “검색 조건이 많아질수록 문자열 템플릿 안에서 분기를 관리하는 비용과 위험이 커졌기 때문입니다. 템플릿 길이와 가독성 문제, 조건 조합의 테스트 어려움이 있었고 변경 영향도 파악도 쉽지 않았습니다. 그래서 필터·정렬·검색 조건을 코드 레벨의 구성 요소로 분리하고 동적으로 조합하도록 전환했습니다. 전환 과정에서는 기존 쿼리와 신규 쿼리의 결과 건수와 상위 결과를 비교해 정합성을 검증했습니다. 워트의 특허 검색식처럼 조건이 복잡한 환경에서도 파싱·검증·DSL 생성을 분리하면 확장성과 테스트 가능성을 높일 수 있다고 생각합니다.”

## Q7. 형태소 사전은 어떻게 운영했나요?

> “사전 자체보다 변경과 배포 과정이 품질을 좌우한다고 봤습니다. 위메프에서는 상품명·브랜드·속성어를 사용자 사전에 반영하고 Nori 분석 결과를 확인했고, 알바천국에서는 동의어·불용어·사용자 사전을 중앙에서 관리해 모든 ES 노드에 동기화하는 Rust 시스템을 만들었습니다. 변경 전후 분석 결과와 대표 검색어를 검증하고, 노드별 파일 불일치와 반영 실패를 확인할 수 있게 했습니다. 그 결과 업데이트 시간을 30분에서 9분으로 줄이고 수동 오류를 크게 낮췄습니다. 특허에서는 전문용어와 약어, 다국어 표현이 더 중요하므로 도메인 전문가의 피드백을 재현 가능한 평가셋과 사전 변경 이력으로 관리하겠습니다.”

## Q8. 검색 품질 개선을 어떻게 측정합니까?

> “먼저 검색 목적에 맞는 오프라인 평가와 온라인 지표를 구분합니다. 오프라인에서는 대표 질의와 관련 문서의 정답셋을 만들고 Precision, Recall, MRR, NDCG 같은 지표를 봅니다. 온라인에서는 CTR, 검색 후 행동, 무결과율, 재검색률, 응답시간과 비즈니스 지표를 함께 봅니다. 채용관 랭킹에서는 필터 조건을 지키면서 광고 유형별 가중치를 적용하고 노출량과 응답시간을 확인했습니다. 워트의 특허 검색은 누락 비용이 클 수 있으므로 일반 커머스보다 Recall과 검색식 재현성이 더 중요할 수 있고, 변리사·IP 전문가와 평가 기준을 합의하는 것이 핵심이라고 생각합니다.”

## Q9. 색인 파이프라인의 정합성은 어떻게 보장합니까?

> “원천 데이터와 색인 문서 사이의 추적 가능성을 먼저 확보합니다. 문서 ID와 변경 시각, 버전 또는 이벤트 식별자를 기준으로 중복·누락을 확인하고, 정적 색인과 증분 색인의 책임을 나눕니다. 대규모 변경은 새 인덱스에 색인한 뒤 건수, 주요 필드, 샘플 쿼리 결과를 비교하고 alias를 전환하는 방식을 선호합니다. 실패 이벤트는 재처리할 수 있어야 하고, Kafka를 사용한다면 Consumer Lag과 실패 큐를 관찰해야 합니다. 위메프에서 Java/JPA 기반 정적·증분 색인을 운영하고 Linux 이관 때 데이터 정합성을 검증했던 경험을 워트의 특허 인덱싱에도 적용할 수 있습니다.”

## Q10. Kafka→Vector 전환에서 무엇을 배웠나요?

> “중간 구성 요소를 줄이는 것 자체보다 처리 요구에 맞는 도구를 선택하는 것이 중요하다는 점을 배웠습니다. 기존 Logstash 구조의 메모리 사용량과 처리량을 측정한 뒤, 필요한 필터·변환을 Vector로 옮기고 버퍼와 재시도 정책을 조정했습니다. 벤치마크 결과 메모리는 약 40% 줄고 처리 속도는 약 60% 향상됐습니다. 전환 후 대시보드와 실패 관측도 함께 재구성했습니다. 워트에서도 기술 선택을 유행으로 결정하기보다 데이터량, 변환 복잡도, 장애 복구, 운영 인력까지 포함해 판단하겠습니다.”

## Q11. 특허 검색식 파서를 만든다면 어떤 구조로 설계하겠습니까?

> “Lexer, Parser, AST, 의미 검증, Query DSL 변환을 분리하겠습니다. AND·OR·NOT, 괄호, 필드 지정, 구문 검색, 근접 연산 같은 문법을 명시하고 우선순위를 문법에 반영하겠습니다. 파싱 오류는 단순 실패가 아니라 사용자가 수정할 위치와 이유를 알려줘야 합니다. AST 단계에서는 존재하지 않는 필드, 허용하지 않는 연산 조합, 과도하게 무거운 표현을 검증하고, 마지막에 ES 또는 OpenSearch 쿼리로 변환하겠습니다. 직접 특허 검색식 파서를 운영한 경험은 없지만, 복잡한 검색 조건을 코드 기반 DSL로 구조화한 경험과 수학 전공에서 익힌 논리적 구조화 역량을 활용할 수 있습니다.”

## Q12. Vector Search와 RAG를 어떻게 바라보나요?

> “기존 키워드 검색을 대체하는 기술이라기보다 서로의 약점을 보완하는 구성으로 봅니다. 특허 검색은 정확한 용어·필드·Boolean 조건과 높은 재현성이 중요하므로 lexical 검색이 여전히 핵심이고, 표현이 다른 유사 기술 탐색에는 vector 검색이 도움이 됩니다. 두 결과를 hybrid retrieval로 결합하고 reranking한 뒤, RAG에서는 답변이 어떤 특허 문헌과 문장에 근거했는지 추적 가능해야 합니다. 저는 운영 경험이 강점이기 때문에 모델 선택뿐 아니라 인덱싱 지연, 벡터 버전, 성능 비용, 품질 평가와 장애 시 fallback까지 함께 보겠습니다.”

## Q13. 왜 Rust를 사용했나요?

> “모든 문제에 Rust를 쓰기보다 안정적으로 오래 실행되는 운영 도구에 적합하다고 판단해 사용했습니다. 사전 동기화, 템플릿 배포 CLI, 로그·메트릭 수집, 개인정보 파기 스케줄러처럼 메모리 안정성과 예측 가능한 성능, 단일 바이너리 배포가 유리한 작업에 적용했습니다. 반면 기존 색인 파이프라인은 Java/JPA 구조를 유지했습니다. 워트에서도 언어 자체를 주장하기보다 팀의 기존 스택과 유지보수 비용을 우선하고, Rust가 명확한 이점을 주는 내부 도구나 고성능 처리 구간에서 제안하겠습니다.”

## Q14. 입사 후 첫 3개월 동안 무엇을 하겠습니까?

> “첫 달에는 검색 구조와 데이터 흐름, 장애 이력, 핵심 지표를 이해하겠습니다. 특히 ES8·OpenSearch 이전 범위, 특허 검색식과 Analyzer, 정적·증분 색인 구조를 문서화하겠습니다. 두 번째 달에는 대표 쿼리셋과 정합성·성능 기준을 만들고, 모니터링이나 배포에서 빠르게 줄일 수 있는 운영 부채 하나를 개선하겠습니다. 세 번째 달에는 작은 범위의 마이그레이션 또는 성능 개선을 직접 완료하고 결과와 롤백 절차를 공유하겠습니다. 먼저 구조를 이해하고, 작게 증명한 뒤 범위를 넓히겠습니다.”

# 5. CEO 예상 질문 및 답변

## Q1. 왜 워트인텔리전스에 지원했나요?

> “제가 쌓아온 검색 운영 경험을 더 깊고 어려운 데이터 문제에 적용할 수 있는 회사라고 판단했기 때문입니다. 지금까지 상품과 채용 검색에서 Analyzer, 사전, 색인, 랭킹, 클러스터 운영을 경험했습니다. 워트는 3억 건 규모의 글로벌 특허 데이터와 전문 검색식, 다국어, AI 리서치를 함께 다루고 있어 검색엔지니어로서 한 단계 더 성장할 수 있는 환경입니다. 동시에 회사가 실제로 필요로 하는 ES8·OpenSearch 이전, 모니터링, 장애 대응, 데이터 인덱싱은 제가 바로 기여할 수 있는 영역입니다. 배우기만 하는 지원이 아니라 기존 경험으로 초기 성과를 만들면서 특허 검색 전문가로 성장하고 싶어 지원했습니다.”

## Q2. 현재 회사에서 이직하려는 이유는 무엇인가요?

> “현재 회사에 대한 불만보다는 제 커리어의 다음 방향이 더 분명해졌기 때문입니다. 알바천국에서 검색 구조 개선과 클러스터 이관, 사전·배포 자동화를 하면서 검색엔지니어로 계속 성장하고 싶다는 확신이 생겼습니다. 다음 환경에서는 더 큰 데이터 규모와 복잡한 검색 문제, AI 검색과 글로벌 데이터까지 다뤄보고 싶습니다. 워트는 검색이 보조 기능이 아니라 제품 경쟁력의 중심이고, 제가 해온 운영 경험도 직접적으로 필요합니다. 지금까지의 경험을 가장 자연스럽게 확장할 수 있는 선택이라고 생각했습니다.”

## Q3. 회사 선택 기준은 무엇인가요?

> “첫째는 기술이 실제 제품 가치의 중심인지, 둘째는 제가 가진 경험으로 기여하면서 더 깊게 성장할 수 있는지, 셋째는 문제를 투명하게 공유하고 개선할 수 있는 문화인지 봅니다. 워트는 특허 데이터 검색과 분석이 사업의 핵심이고, 채용 공고에도 레거시 고도화와 장애 재발 방지를 구체적으로 적어 두었습니다. 공식 컬처의 ‘문제는 구조화부터’, ‘진짜 원인 찾기’도 제가 운영 업무에서 중요하게 생각해 온 방식과 맞습니다. 단순히 새 기술을 많이 쓰는 회사보다 해결할 문제가 분명하고 그 결과가 고객 가치로 연결되는 회사를 선택하고 싶습니다.”

## Q4. 검색엔지니어가 사업에 어떻게 기여할 수 있다고 생각하나요?

> “검색 결과의 신뢰도와 응답 안정성을 고객이 느끼는 제품 가치로 연결해야 한다고 생각합니다. 특허 검색은 한 건을 놓치는 것이 의사결정과 권리 판단에 영향을 줄 수 있고, 결과가 느리거나 불안정하면 아무리 좋은 AI 모델도 사용하기 어렵습니다. 검색엔지니어는 데이터 최신성, Recall, 응답시간, 장애율을 관리해 고객이 안심하고 제품을 쓰게 만드는 역할입니다. 저는 운영 자동화와 모니터링으로 안정성을 높이고, 사전·Analyzer·쿼리 개선으로 검색 품질을 높여 제품 신뢰와 사용 효율에 기여할 수 있습니다.”

## Q5. 특허 도메인을 잘 모르는데 괜찮을까요?

> “현재 특허 실무 전문가는 아니지만, 도메인을 모른다는 점을 가볍게 보지는 않습니다. 검색 품질은 기술만으로 정할 수 없고 사용자가 무엇을 관련 문서로 보는지 이해해야 하기 때문입니다. 입사하면 실제 검색식, 무결과·재검색 사례, 고객 문의, 특허 전문가의 판단 기준을 우선 학습하겠습니다. 위메프에서도 상품명·브랜드·속성어를 이해해야 사전을 개선할 수 있었고, 알바천국에서도 채용 조건과 광고 정책을 이해해야 랭킹을 설계할 수 있었습니다. 새로운 도메인을 데이터와 운영 사례를 통해 학습해 온 방식으로 빠르게 적응하겠습니다.”

## Q6. 3~5년 뒤 어떤 엔지니어가 되고 싶나요?

> “대규모 검색 시스템의 품질과 운영을 함께 책임질 수 있는 검색엔지니어가 되고 싶습니다. 단기적으로는 워트의 레거시 고도화와 OpenSearch 이전을 안정적으로 수행하고, 중기적으로는 특허 검색식·다국어 Analyzer·hybrid search의 품질 평가 체계를 깊게 이해하고 싶습니다. 장기적으로는 검색팀과 데이터·AI·제품팀 사이에서 기술 결정을 연결하고, 장애와 운영 부채를 줄이는 기준을 만드는 역할까지 맡고 싶습니다. 직함보다 회사가 검색과 데이터 문제를 맡길 때 신뢰할 수 있는 사람이 되는 것이 목표입니다.”

## Q7. 빠르게 변하는 조직에서 일할 수 있나요?

> “변화 자체보다 우선순위와 완료 기준을 먼저 맞추는 것이 중요하다고 생각합니다. 운영 환경에서는 긴급 장애와 계획된 개선이 자주 충돌했기 때문에 영향도, 긴급도, 복구 가능성을 기준으로 순서를 정해 왔습니다. 예를 들어 사전 관리나 템플릿 배포처럼 반복적으로 시간을 쓰고 사고가 나던 업무는 자동화해 팀의 여유를 만들었습니다. 워트에서도 모든 것을 한꺼번에 완벽하게 만들기보다 목표와 검증 기준을 합의하고 작은 단위로 배포하면서 피드백을 반영하겠습니다.”

## Q8. 본인이 회사에 줄 수 있는 가장 확실한 가치는 무엇인가요?

> “검색 시스템을 안정적으로 운영하면서도 개선을 멈추지 않는 실행력입니다. 저는 장애 대응만 하거나 기능 개발만 한 것이 아니라, 장애가 늦게 발견되는 문제는 모니터링으로, 수동 배포 충돌은 Git·Jenkins 자동화로, 템플릿 구조의 한계는 Query DSL 전환으로 해결했습니다. 워트에서도 당장 필요한 마이그레이션과 운영 안정화에 기여하면서, 반복되는 문제를 구조적으로 줄이는 엔지니어가 되겠습니다.”

## Q9. 왜 우리가 신승환 님을 뽑아야 하나요?

> “워트가 지금 필요로 하는 검색 운영 경험과 변화 실행 경험을 함께 갖고 있기 때문이라고 생각합니다. Elasticsearch 클러스터와 Kafka 파이프라인을 운영했고, 형태소 사전과 색인, 검색 쿼리와 랭킹까지 다뤘습니다. 무엇보다 Windows→Linux, Mustache→Query DSL, Logstash→Vector처럼 운영 중인 시스템을 검증하며 바꿔 본 경험이 있습니다. 특허와 OpenSearch는 더 배워야 하지만, 낯선 시스템을 빠르게 구조화하고 안정적으로 개선한 경험으로 초기에 기여할 수 있습니다.”

# 6. 컬처핏·협업 질문 및 답변

## Q1. 협업 중 의견 충돌이 생기면 어떻게 하나요?

> “사람의 선호보다 해결하려는 문제와 판단 기준을 먼저 맞춥니다. 예를 들어 기술 전환을 논의할 때 ‘어떤 도구가 더 좋은가’보다 현재 병목이 메모리인지 처리량인지 운영 복잡도인지부터 확인합니다. Kafka→Vector 전환도 기존 구조를 바로 부정하지 않고 측정 결과와 벤치마크로 제안했습니다. 충분히 토론한 뒤 결정되면 제 안이 아니더라도 실행에 맞추고, 결과를 다시 데이터로 확인합니다. 워트가 말하는 ‘논쟁하되 따르기’와 비슷한 방식이라고 생각합니다.”

## Q2. 비개발자나 다른 팀과는 어떻게 소통하나요?

> “기술 용어보다 영향과 선택지를 중심으로 설명합니다. 예를 들어 검색 템플릿 문제를 설명할 때 템플릿 문법보다 변경 충돌, 책임자 추적 불가, 배포 시간 같은 팀의 문제로 설명했습니다. 이후 Git·Jenkins 자동화가 어떤 위험을 줄이는지와 배포 절차가 어떻게 바뀌는지를 공유했습니다. 검색 품질 문제도 Analyzer 설정부터 말하기보다 어떤 검색어에서 어떤 결과가 누락되는지를 보여주는 편입니다. 상대가 결정해야 할 내용과 제가 책임질 기술 내용을 구분해서 소통합니다.”

## Q3. 요구사항이 모호하면 어떻게 합니까?

> “바로 구현하기보다 왜 필요한지, 누가 쓰는지, 성공 기준이 무엇인지부터 확인합니다. 검색에서는 ‘정확도를 높여 달라’는 말만으로는 개발할 수 없기 때문에 대표 질의와 기대 결과, 절대 빠지면 안 되는 조건, 허용 가능한 응답시간을 구체화합니다. 그다음 작은 샘플로 결과를 보여주고 방향을 확인합니다. 완벽한 요구사항을 기다리기보다 불확실성을 질문과 실험으로 줄이겠습니다.”

## Q4. 여러 업무의 우선순위는 어떻게 정하나요?

> “사용자 영향, 긴급도, 재발 가능성, 복구 가능성을 기준으로 정합니다. 장애와 데이터 누락은 가장 먼저 처리하고, 반복적으로 사고를 만드는 수동 업무는 자동화 우선순위를 높입니다. 반면 구조 개선은 한 번에 크게 바꾸기보다 운영 일정과 롤백 가능성을 고려해 나눕니다. 우선순위가 충돌하면 제가 임의로 숨겨서 조정하기보다 각 선택의 영향과 예상 시간을 공유하고 담당자와 합의합니다.”

## Q5. 실수하거나 실패한 경험을 어떻게 다루나요?

> “실수를 숨기지 않고 영향 범위를 먼저 확인한 뒤 복구와 재발 방지를 분리해서 처리합니다. 운영에서는 개인이 조심하는 것만으로 같은 문제가 사라지지 않는다고 생각합니다. 그래서 수동 사전 배포나 검색 템플릿 배포처럼 오류가 반복될 수 있는 과정은 검증과 이력 추적이 가능한 시스템으로 바꿨습니다. 면접에서는 실제 경험한 실수 한 건을 골라 제가 놓친 신호, 복구 과정, 이후 추가한 체크나 자동화를 구체적으로 말씀드리겠습니다.”

> 준비 메모: 실제 장애·실수 사례 1건의 상황, 본인 판단, 영향, 복구 시간, 재발 방지를 반드시 채운다.

## Q6. 피드백을 받는 방식은 어떤가요?

> “의도와 사실을 분리해 듣고, 바로 방어하기보다 상대가 우려하는 결과를 확인합니다. 검색 결과나 운영 방식은 정답이 한 번에 나오기 어렵기 때문에 코드 리뷰, 지표, 사용자 피드백을 받아 수정하는 것이 자연스럽다고 생각합니다. 반대로 피드백을 줄 때도 사람에 대한 평가보다 재현 조건과 영향, 개선 제안을 함께 전달합니다.”

## Q7. 새로운 분야를 어떻게 학습하나요?

> “업무에서 바로 검증할 작은 문제를 정하고 공식 문서, 실험, 운영 로그 순서로 학습합니다. Rust도 문법만 공부한 것이 아니라 사전 동기화나 모니터링처럼 실제 운영 문제에 적용하며 익혔습니다. 특허 도메인도 용어집만 외우기보다 대표 검색식과 결과, 고객 사례, 무결과·오탐 사례를 보면서 학습하겠습니다. 모르는 것을 빨리 드러내고 도메인 전문가에게 구체적으로 질문하는 편입니다.”

## Q8. 본인이 선호하는 팀 문화는 무엇인가요?

> “문제를 투명하게 공유하고, 근거를 가지고 토론하며, 결정 후에는 함께 실행하는 문화입니다. 운영 문제를 개인의 주의력으로만 해결하면 같은 문제가 반복되기 때문에 원인과 변경 이력을 남길 수 있어야 합니다. 워트가 공개한 ‘문제는 구조화부터’, ‘진짜 원인 찾기’, ‘먼저 맞추고 시작하기’, ‘논쟁하되 따르기’는 제가 일할 때 중요하게 생각하는 방식과 잘 맞습니다.”

## Q9. 주도적으로 개선한 경험을 말해 주세요.

> “검색 템플릿 배포 과정의 자동화 경험이 대표적입니다. 수작업 배포로 충돌이 발생하고 누가 무엇을 바꿨는지 추적하기 어려웠습니다. Rust CLI로 Git의 템플릿을 파싱해 Elasticsearch `_scripts` API로 배포하고, 개발·운영 브랜치와 Jenkins 파이프라인을 연결했습니다. 배포 시간이 90% 이상 줄었고 변경 이력과 책임자를 추적할 수 있게 됐습니다. 요청받은 기능만 만드는 것보다 팀이 반복해서 겪는 문제를 구조적으로 줄인 경험입니다.”

# 7. 검색엔지니어 직무 질문 및 답변

## Q1. Precision과 Recall 중 무엇이 더 중요합니까?

> “서비스 목적에 따라 다르며 특허 검색에서는 Recall의 비용을 특히 크게 봐야 한다고 생각합니다. 선행기술 문헌을 놓치면 중요한 판단에 영향을 줄 수 있기 때문입니다. 다만 Recall만 높이면 검토량이 과도해지므로 1차 검색은 누락을 줄이고, 필드 가중치·분류·reranking으로 상위 결과의 Precision을 높이는 단계적 구조가 적합합니다. 최종 기준은 IP 전문가와 질의 유형별로 합의해야 합니다.”

## Q2. Nori와 사용자 사전을 운영하면서 무엇을 중요하게 봤나요?

> “분석기 설정 자체보다 실제 검색어가 어떤 토큰으로 분해되고 그 결과가 검색 결과에 어떤 영향을 주는지 봤습니다. 상품명·브랜드·복합명사처럼 잘못 분해되는 표현을 사용자 사전에 반영하고, 변경 전후 `_analyze` 결과와 대표 질의 결과를 비교했습니다. 사전이 계속 커지면 중복과 부작용이 생길 수 있어 변경 이유, 담당자, 테스트 검색어를 함께 관리하는 것이 중요합니다.”

## Q3. 동의어와 사용자 사전의 차이는 어떻게 판단합니까?

> “사용자 사전은 형태소가 어떻게 분해되는지를 바로잡는 데 사용하고, 동의어는 서로 다른 표현을 같은 검색 의도로 연결할 때 사용합니다. 예를 들어 전문 복합어가 잘못 분해되면 사용자 사전 문제이고, 약어와 정식 명칭을 연결하는 것은 동의어 문제에 가깝습니다. 둘을 섞어 해결하면 토큰 폭증이나 의도하지 않은 매칭이 생길 수 있어 분석 단계와 검색 확장 단계를 구분합니다.”

## Q4. 무거운 검색 쿼리는 어떻게 최적화합니까?

> “먼저 slow log와 profile API로 어느 절이 시간을 쓰는지 확인합니다. scoring이 필요 없는 조건은 filter context로 옮기고, wildcard·regexp·script sort·대규모 aggregation·깊은 pagination 사용을 점검합니다. Mapping과 doc_values, cardinality, shard 수, 요청 캐시도 함께 봅니다. 쿼리 하나만 수정하기보다 동일 기능을 더 싼 방식으로 제공할 수 있는지 제품 요구와 함께 논의합니다.”

## Q5. 샤드 수는 어떻게 정합니까?

> “문서 수만으로 정하지 않고 인덱스 크기, 노드 수, 검색·색인 부하, 보관 기간, 장애 복구 시간을 함께 봅니다. 너무 많은 작은 샤드는 heap과 cluster state 부담을 만들고, 너무 큰 샤드는 재배치와 복구 시간이 길어집니다. 예상 데이터 증가량을 바탕으로 초기값을 잡고 실제 shard 크기와 query latency, merge 비용을 보며 조정합니다. rollover 가능한 시계열 인덱스와 검색용 장기 인덱스는 전략을 다르게 가져갑니다.”

## Q6. GC가 증가하면 어떻게 대응합니까?

> “heap 사용 패턴과 old GC 시간, 빈도, 검색 지연의 상관관계를 먼저 봅니다. 과도한 aggregation, fielddata, 큰 result window, shard 과다, 색인 buffer, 캐시 사용을 확인합니다. 단순히 heap만 늘리기보다 객체를 많이 만드는 쿼리와 shard 구조를 줄이고, OS page cache를 해치지 않는 범위에서 heap을 설정해야 합니다. 긴 GC가 사용자 지연으로 이어지기 전에 알림 임계치와 추세를 관리하겠습니다.”

## Q7. 무중단 재색인은 어떻게 합니까?

> “버전이 붙은 새 인덱스를 만들고 정적 색인을 수행한 뒤, 증분 변경분을 따라잡게 합니다. 원본과 신규 인덱스의 건수, 주요 필드, 샘플·대표 쿼리 결과와 성능을 비교합니다. 기준을 통과하면 alias를 원자적으로 전환하고, 문제 발생 시 기존 alias로 되돌릴 수 있게 일정 기간 이전 인덱스를 유지합니다. 쓰기 경로가 있다면 이중 쓰기 또는 변경 로그 재처리 전략을 명확히 해야 합니다.”

## Q8. Mapping 충돌이나 색인 누락은 어떻게 탐지합니까?

> “색인 실패를 애플리케이션 로그에만 남기지 않고 실패 건수와 원인을 별도로 수집해야 합니다. 동적 Mapping 의존도를 낮추고 필드 타입을 명시하며, 배포 전에 샘플 데이터로 호환성을 검사합니다. 원천 건수와 색인 건수, 최신 변경 시각, 실패 큐를 비교하고 문서 ID 기준 샘플 검증을 수행합니다. Linux 이관 때도 Mapping 충돌과 정적·증분 색인 누락을 별도 체크리스트로 확인했습니다.”

## Q9. 검색 랭킹을 개선한 경험을 설명해 주세요.

> “알바천국 채용관에서 지역·직종·근무조건 같은 필터를 유지하면서 유료공고 유형별 가중치를 적용했습니다. `function_score`, `boost`, `random_score`를 조합해 노출 정책을 구조화하고 Kibana에서 결과를 검증했습니다. 결과적으로 유료공고 노출량은 평균 25% 증가하고 응답시간도 최대 15% 개선됐습니다. 중요한 점은 광고 가중치가 사용자 검색 조건을 침범하지 않도록 필터와 scoring의 책임을 분리한 것입니다.”

## Q10. 검색 결과 정합성은 어떻게 검증합니까?

> “대표 질의를 유형별로 나누고 결과 건수, 상위 N개 문서, 스코어 또는 정렬 키, 필수 포함·제외 조건을 비교합니다. 전체 문서를 일일이 비교하기 어렵다면 고빈도 질의, 매출·업무 영향이 큰 질의, 경계 조건을 우선합니다. Query DSL 전환과 클러스터 이관 때도 기존 결과와 신규 결과를 비교하는 방식으로 안정화했습니다. 차이가 생기면 오류인지 의도한 개선인지 제품 담당자와 구분합니다.”

## Q11. 장애 대응 경험에서 가장 중요한 원칙은 무엇인가요?

> “원인 분석보다 먼저 영향 확산을 막고, 복구 뒤에 원인과 재발 방지를 끝까지 책임지는 것입니다. 검색 오류율과 지연, shard·노드 상태를 확인해 범위를 줄이고 최근 변경을 우선 확인합니다. 롤백이나 기능 제한으로 서비스를 먼저 안정화한 뒤 로그와 메트릭으로 원인을 재현합니다. 이후 알림, 자동화, 런북, 테스트 중 하나 이상을 추가해야 장애 대응이 완료됐다고 봅니다.”

## Q12. Elasticsearch와 OpenSearch의 차이를 어떻게 학습하겠습니까?

> “버전 계보에 대한 일반론보다 워트가 실제 사용하는 기능 목록부터 비교하겠습니다. Query DSL, Analyzer·플러그인, security, snapshot, client, index template, ingest pipeline, vector 기능을 표로 만들고 공식 문서와 실험 클러스터에서 확인하겠습니다. 대표 인덱스와 쿼리셋을 옮겨 기능·스코어·성능 차이를 자동 비교하면 학습과 마이그레이션 검증을 동시에 할 수 있습니다.”

## Q13. 다국어 특허 검색 인덱스를 설계한다면?

> “원문을 보존하고 언어별 분석 필드와 정규화·번역 필드를 분리하겠습니다. 언어 감지의 신뢰도, 국가별 필드 구조, 전문용어와 약어, 동일 특허의 패밀리 관계를 고려해야 합니다. 한 필드에 모든 언어를 섞기보다 언어별 Analyzer를 명시하고 cross-field 검색 전략을 세우겠습니다. 정확한 구조는 기존 데이터와 고객 검색식을 먼저 분석한 뒤 결정하겠습니다.”

## Q14. 검색식 오류를 사용자에게 어떻게 보여줘야 하나요?

> “단순히 ‘파싱 실패’라고 하지 않고 오류 위치, 기대한 토큰, 수정 예시를 알려줘야 합니다. 입력 중 자동 완성이나 괄호 짝 검사를 제공할 수 있고, 서버에서는 최대 깊이·절 수·비용이 큰 표현을 제한해 시스템을 보호해야 합니다. 사용자 편의와 클러스터 보호를 파서·검증 단계에서 함께 설계하겠습니다.”

## Q15. 색인과 검색이 동시에 느려지면 무엇을 의심합니까?

> “노드 자원 경쟁, shard·merge, disk I/O, heap·GC, refresh·replica 설정을 우선 봅니다. 대량 색인 시 refresh interval과 replica 조정을 검토할 수 있지만 검색 요구와 장애 허용 수준을 함께 고려해야 합니다. 색인 노드와 검색 노드 역할 분리, bulk 크기와 동시성, merge backlog, disk watermark도 확인합니다. 임시 완화와 근본 원인을 구분해 대응하겠습니다.”

## Q16. 검색 품질과 응답속도가 충돌하면 어떻게 결정합니까?

> “사용자가 체감하는 허용 지연과 품질 개선 폭을 함께 측정합니다. 모든 요청에 비싼 로직을 적용하기보다 질의 유형별로 적용하거나 후보군을 줄인 뒤 reranking할 수 있습니다. timeout과 fallback을 두고, 품질 지표와 P95·P99 지연을 같이 비교합니다. 기술적으로 가능한 최대 품질보다 제품이 안정적으로 제공할 수 있는 최적점을 찾겠습니다.”

# 8. 마지막 질문으로 회사에 물어볼 만한 질문

## 9.2 30초 지원 동기

> “검색이 회사 제품의 중심인 환경에서 더 깊은 검색 문제를 풀고 싶어 지원했습니다. 저는 상품·채용 검색에서 형태소 사전, 색인, Query DSL, 랭킹뿐 아니라 클러스터 이관과 장애 대응까지 경험했습니다. 워트가 추진하는 ES8·OpenSearch 이전과 특허 데이터 인덱싱에는 바로 기여할 수 있고, 다국어·특허 검색과 AI Native Research 영역에서는 장기적으로 성장할 수 있다고 판단했습니다.”

## 9.3 대표 경험 5개 STAR 카드

### 1. Linux 이관

- 상황: Windows ES 클러스터의 안정성·확장성·비용 개선 필요
- 행동: Linux 신규 구축, Mapping·Analyzer·템플릿 호환성, 정적·증분 색인과 데이터 정합성 검증, 모니터링 재구축
- 결과: 데이터 손실 없이 이관, 성능 약 15% 개선, 라이선스 비용 약 66% 절감
- 연결: ES8·OpenSearch 마이그레이션

### 2. 검색 모니터링

- 상황: 이상 징후를 늦게 발견하고 대응에 약 30분 소요
- 행동: Rust 메트릭 수집, Grafana·Kibana 대시보드, 메신저·Telegram 알림
- 결과: 대응 시간 약 10분, 운영 가시성 향상
- 연결: 검색 클러스터 장애 관리와 재발 방지

### 3. 사전 관리 자동화

- 상황: 모든 ES 노드의 사전을 수동 배포해 시간과 오류 발생
- 행동: Rust 기반 중앙 관리·노드 동기화·검증 체계
- 결과: 30분→9분, 수동 오류율 90% 감소
- 연결: 특허 전문용어·다국어 형태소 사전 운영

### 4. Mustache→Query DSL

- 상황: 템플릿 길이, 복잡한 분기, 테스트·유지보수 한계
- 행동: 필터·정렬·조건을 코드 구성 요소로 분리하고 기존 결과와 정합성 검증
- 결과: 확장성·가독성·개발 생산성 향상, 기술 부채 감소
- 연결: 복잡한 특허 검색식과 검색 레거시 고도화

### 5. Kafka→Vector

- 상황: Logstash 구조의 메모리 사용과 처리량 병목
- 행동: 성능 분석, Vector 변환·필터·버퍼 구성, 벤치마크, 모니터링 재구축
- 결과: 메모리 40% 감소, 처리 속도 60% 향상
- 연결: 대용량 특허 데이터 파이프라인 최적화

# 면접장에서 꼭 말해야 할 핵심 메시지 5개

1. **저는 Elasticsearch 사용자가 아니라 검색 서비스 운영자입니다.**  
   클러스터, 색인, Analyzer·사전, 쿼리, 랭킹, 모니터링, 장애 대응을 End-to-End로 경험했습니다.

2. **운영 중인 레거시를 안전하게 바꿔 본 경험이 있습니다.**  
   Windows→Linux, Mustache→Query DSL, Logstash→Vector 전환에서 정합성·성능·롤백을 고려했습니다.

3. **장애를 복구하는 데서 끝내지 않고 재발 구조를 줄였습니다.**  
   모니터링, 알림, Git·Jenkins 배포, 사전 동기화를 통해 대응 시간과 수동 오류를 줄였습니다.

4. **검색 품질과 사업 가치를 함께 봅니다.**  
   형태소·사전·랭킹을 기술 설정이 아니라 사용자의 검색 결과, 운영 효율, 제품 신뢰와 연결해 개선했습니다.

5. **특허·OpenSearch·다국어는 과장하지 않고 빠르게 학습하되, 기존 운영 경험으로 바로 기여하겠습니다.**  
   초기에는 마이그레이션과 운영 안정화에 성과를 만들고, 장기적으로 특허 검색과 hybrid search까지 책임지겠습니다.
