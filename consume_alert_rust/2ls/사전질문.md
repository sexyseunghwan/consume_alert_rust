이 프로젝트에서 가장 중요하게 본 부분은 단순 서버 이전이 아니라 검색 결과 정합성과 데이터 무결성이었습니다. 기존 검색 템플릿과 신규 환경의 결과를 비교하고, 색인 누락이나 매핑 충돌이 없는지 검증하면서 서비스 안정성을 우선으로 이관을 진행했습니다.

정량적 성과로는 데이터 손실 없이 클러스터 이관을 완료했고, 디스크 I/O 및 리소스 최적화를 통해 검색 성능을 약 15% 개선했습니다. 또한 Windows에서 Linux 환경으로 전환하면서 서버 라이선스 비용을 약 66% 절감했습니다. 이 경험은 워트인텔리전스의 ES8 이전, OpenSearch 기능 이전, 검색 레거시 고도화 업무와 직접적으로 연결될 수 있다고 생각합니다.

특히 반복적으로 발생하는 운영 문제를 단순 수작업으로 처리하지 않고, 자동화와 모니터링 체계로 바꾸는 데 강점이 있습니다. 예를 들어 검색 템플릿 배포를 Rust CLI와 Jenkins/Git 기반으로 자동화해 배포 시간을 90% 이상 단축했고, Elasticsearch 사전 업데이트도 중앙 관리 방식으로 개선해 업데이트 시간을 30분에서 9분으로 줄였습니다.

4.협업 갈등 해결 경험 *

검색 템플릿 배포 프로세스를 개선하는 과정에서 협업 방식에 대한 의견 차이가 있었습니다.

기존에는 Elasticsearch Mustache Template을 수작업으로 배포하는 방식이었기 때문에 변경 속도는 빠를 수 있었지만, 배포 충돌, 변경 이력 관리 부재, 책임자 추적 불가 문제가 있었습니다. 서비스 개발 입장에서는 빠른 반영이 중요했고, 검색 운영 입장에서는 안정성과 이력 관리가 중요했기 때문에 어느 쪽 기준을 우선할지에 대한 관점 차이가 있었습니다.

저는 먼저 문제를 개인의 실수나 특정 팀의 방식으로 보지 않고, “수작업 배포 구조 자체가 충돌과 누락을 만들 수밖에 없는 구조”라고 정리했습니다. 이후 Git 기반 변경 이력 관리, 개발/운영 브랜치 분리, Jenkins Pipeline을 통한 자동 배포 구조를 제안했고, 서비스 개발자는 파라미터 전달만으로 검색 기능을 연동할 수 있도록 프로세스를 단순화했습니다.

결과적으로 템플릿 배포 시간이 수작업 대비 90% 이상 단축되었고, 변경 이력과 책임자 추적이 가능해졌으며, 운영 배포 충돌 리스크도 줄일 수 있었습니다. 이 경험을 통해 협업 갈등은 한쪽 의견을 이기는 방식보다, 각자의 우려를 구조적으로 해결하는 방식이 가장 효과적이라는 것을 배웠습니다.

5. 자기 주도적으로 일을 만들어낸 경험 *

알바천국에서 Elasticsearch 사전 관리 자동화 시스템을 만든 경험이 있습니다.

이 문제를 보고 단순히 작업을 더 조심해서 처리하는 방식으로는 한계가 있다고 판단했습니다. 그래서 Rust 기반으로 중앙 집중식 사전 관리 시스템을 설계하고, master 서버에서 변경된 사전을 모든 Elasticsearch 노드에 자동 동기화하는 구조를 구현했습니다. 운영 환경 테스트를 거쳐 실제 배포까지 진행했고, 사전 변경 후 반영 여부를 더 쉽게 확인할 수 있도록 운영 프로세스도 함께 정리했습니다.
